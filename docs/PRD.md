## 1. 背景与目标

### 1.1 问题陈述
城市规划与建筑设计院日常工作高度依赖两类知识：
- **外部规范**：规划法规、技术标准、设计指引、用地分类、停车/公服设施标准等；
- **内部资料**：企业制度、流程指引、报销/IT 行政，以及大量**按项目隔离**的项目资料（控规、城市设计、TOD、报审清单等）。

现状痛点：
1. **查得慢**：规范条文分散在大量 PDF/Word 中，关键指标（如容积率、退线、配建比）常在表格里，关键词搜索命中差。
2. **易出错**：人工转述条文易遗漏前提条件，AI 通用问答会编造不存在的条款。
3. **权限乱**：项目资料应按项目/角色隔离，但传统网盘做不到「检索时」就过滤，存在越权查看风险。

### 1.2 产品目标
构建一个**可信、可控、可隔离**的企业 RAG 问答平台：
- **可信**：结论只来自检索命中的原文，依据不足即拒答，杜绝编造。
- **可控**：表格作为一级对象由程序精确渲染，LLM 不手写表格数据。
- **可隔离**：检索发生前完成权限过滤，无权资料不进入 LLM 上下文，更不出现在引用中。

### 1.3 衡量指标（北极星 & 辅助）
| 指标 | 定义 | 目标 |
| --- | --- | --- |
| 评测通过率 | `/evaluation` 题库批量运行的通过比例 | 持续提升，回归不下降 |
| 拒答准确率 | 应拒答场景正确拒答、应作答场景正确作答 | 双向都不误判 |
| 越权泄露率 | 无权资料出现在回答/引用中的比例 | **0** |
| 引用可溯源率 | 结论可对应到具体文件/章节/页码 | 100% |

## 2. 用户与角色

四类角色 × 三级权限（公开 L1 / 项目主管 L2 / 管理员 L3），定义见 `lib/knowledge/permissions.ts`。

| 角色 | ID | 最高权限级 | 可访问范围 |
| --- | --- | --- | --- |
| 普通员工 | `employee` | L1 | L1 公开资料 + 被授权（`projectIds`）的项目资料 |
| 项目负责人 | `project_manager` | L2 | L1 + L2 非项目资料 + 自己负责（`ownedProjectIds`）的项目资料 |
| 管理员 | `admin` | L3 | 全部资料 + 文档管理 |
| 开发人员 | `developer` | L3 | 全部资料 + 检索调试工具 |

> 项目资料通过 `projectId / projectOwnerId / accessibleUserIds` 三种方式绑定可见性；
> 同时满足「`enabled` 且 `status=indexed`」才会进入检索。

**用户旅程（核心场景）**：员工在问答工作台输入问题 → 系统按其角色过滤可见资料 → 命中则返回【结论/依据/注意】结构化回答 + 引用卡片（含表格切片）→ 命中无权资料则只提示权限不足、不泄露原文 → 依据不足则明确拒答。

## 3. 功能需求

### 3.1 问答工作台（`/`）—— P0
- **FR-1.1** 支持切换模拟账号（普通员工/项目负责人/管理员；开发账号当前隐藏）。
- **FR-1.2** 输入自然语言问题，可附带城市维度。
- **FR-1.3** 返回结构化回答：`【结论】`（LLM 抽取式）/`【依据】`（引用卡片）/`【注意】`（适用范围声明）。
- **FR-1.4** 命中表格时，回答内嵌**表格切片**（真实行列由程序渲染，高亮命中行）。
- **FR-1.5** 给出置信度标签（高/中/低，依据条数决定）。
- **FR-1.6** 拒答场景输出 `【无法确定】/【原因】/【建议】` 三段式。

### 3.2 文档管理（`/documents`）—— P0
- **FR-2.1** 上传 PDF / Word 文档。
- **FR-2.2** 触发处理流水线：解析 → 切片 → embedding → 入库。
- **FR-2.3** 配置文档分类（14 类文件类型）、权限级别、项目归属、可见人员。
- **FR-2.4** 切换检索开关（`enabled`）、删除文档。
- **FR-2.5** 列表按当前用户的项目 ACL 过滤；变更操作限文档所属负责人/管理员。

### 3.3 切片查看器（`/chunks`）—— P1
- **FR-3.1** 查看文档的知识单元（Chunk）与 14 类结构化知识对象提取结果。

### 3.4 检索调试（`/debug`）—— P1（开发角色可见）
- **FR-4.1** 展示关键词提取、精确索引/BM25/向量三路得分、重排过程。

### 3.5 题库评测（`/evaluation`）—— P0
- **FR-5.1** 导入题库（CSV / TSV / Markdown 表格 / XLSX，自动识别列）。
- **FR-5.2** 批量运行评测，覆盖项目权限/拒答/企业制度等场景。
- **FR-5.3** 展示通过率，导出报告（XLSX）。
- **FR-5.4** 每题可绑定模拟账号，验证角色相关行为。

### 3.6 反馈（`/api/feedback`）—— P1
- **FR-6.1** 对回答/拒答记录反馈，按 `feedbackTargetId` 存储。

## 4. RAG 处理链路（核心业务流）

### 4.1 文档入库流水线
```
上传(PDF/Word)
  → 文本提取 (pdfjs-dist / pdfplumber Python 边车 / 扫描件 OCR·GLM-4V 兜底)
  → IR 中间表示 (Block[]: 标题/段落/列表/表格)
  → 文档画像 DocProfile
  → 14 类知识对象提取 (条款/指标/定义/成果要求/流程步骤/用地代码…)
  → 切片 Chunk[] + 独立表格对象 RagTable[]
  → embedding 入库
```

### 4.2 问答链路（6 道有序闸门，见 `lib/rag/generateAnswer.ts`）
```
1. 检索前范围判断  checkScope()        —— 超范围 → 拒答
2. 混合检索        retrieve()          —— 精确索引 + BM25 + 向量，多维重排
3. 权限闸门        shouldReturnNoAccess() —— 只命中无权资料 → 权限提示，不泄露原文
4. 检索后依据判断  checkEvidence()     —— 依据不足 → 拒答
5. LLM 抽取式生成  synthesizeConclusion() —— 仅基于 Top-5 chunk；检测 LLM 自拒答
6. 表格装配 + 拼装 assembleTableSlices() —— 程序渲染表格，输出结构化回答 + 引用
```

**关键不变量**（实现约束）：
- `LLM_CONTEXT = 5` 且 `MAX_CITATIONS ≥ LLM_CONTEXT` —— 避免「结论基于第 5 条、引用却不展示它」的矛盾。
- 全程不允许编造：结论来自 LLM 抽取的 chunk 原文，引用来自检索命中的 chunk。

## 5. 非功能需求

| 类别 | 要求 |
| --- | --- |
| 可演示性 | 不配置 AI Key 时使用内置 mock LLM / mock Embedding，全链路可跑通 |
| 安全 | 权限过滤发生在检索前；无权资料不进入 LLM 上下文与引用 |
| 可溯源 | 每条引用含文件名/章节/条款/页码/原文片段 |
| 可扩展 | AI 走 OpenAI 兼容接口（默认接智谱 GLM），可替换 |
| 可验证 | `npm test`（74 例）+ `tsc --noEmit` + `npm run build` 三件套 |
| 平台 | Web（Next.js 14），默认端口 3100 |

## 6. 验收标准（与评测题库一一对应）

| 编号 | 场景 | 期望行为 | 验证方式 |
| --- | --- | --- | --- |
| AC-1 | 命中公开资料 | 返回结构化结论 + 引用 | 评测题库 + 单测 |
| AC-2 | 命中表格类问题 | 内嵌真实表格切片、高亮命中行 | 评测题库 |
| AC-3 | 仅命中无权资料 | 返回权限提示，**不泄露原文/引用** | `无权项目资料命中时返回权限提示且不泄露引用` |
| AC-4 | 依据不足 / 超范围 | 三段式拒答，不编造 | `checkScope` / `checkEvidence` 用例 |
| AC-5 | 角色越权 | 按 ACL 正确过滤 | `splitChunksByUserAccess` 系列用例 |
| AC-6 | 文档管理权限 | 仅负责人/管理员可变更 | `document management mutations…` 用例 |

## 8. 开放问题 / 风险
- **R1** 表格抽取依赖 Python（pdfplumber）边车，缺失会降级 —— 部署需保证 Python 可用。
- **R2** mock LLM 逐字复述原文，自拒答检测需谨防误杀（已只匹配首句）。
- **R3** 当前无持久化，重启丢数据，仅适合演示/评测，不可直接上生产。
