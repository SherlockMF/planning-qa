## 1. 为什么要评测

本产品的核心承诺是「**可信、可控、可隔离**」，这些都是**行为正确性**而非「答得流畅」。
因此必须用可量化的题库回归来守住三件事：
1. 该作答的正确作答（不漏答）；
2. 该拒答的正确拒答（不编造）；
3. 无权资料绝不泄露（隔离不破）。

评测既是质量门，也是迭代护栏——任何检索/重排/拒答逻辑改动都要先过题库。

---

## 2. 两层评测

| 层 | 载体 | 触发 | 用途 |
| --- | --- | --- | --- |
| **代码级单测** | `tests/*.test.ts`（`npm test`） | CI/本地，每次改动 | 权限、解析、导入导出等单元正确性，74 例 |
| **题库级评测** | `/evaluation` 页面 + `lib/rag/eval/` | 手动/批量运行 | 端到端问答行为：作答/拒答/权限，按通过率衡量 |

两层互补：单测锁住「函数行为」，题库锁住「整条问答链路在真实问题上的表现」。

## 3. 场景覆盖矩阵

题库需覆盖以下场景（对应单测 `evaluationEnterpriseSamples` / `knowledgePermissions` 已固化样本）：

| 场景类别 | 示例问题 | 模拟账号 | 期望行为 |
| --- | --- | --- | --- |
| 公开规范作答 | 二类居住用地容积率上限 | 任意 | 作答 + 引用（文件/页码） |
| 表格类指标 | >90㎡ 住宅每户停车位标准 | 任意 | 作答 + 表格切片高亮命中行 |
| 项目资料-有权 | 滨江控规某指标 | 李婷（负责该项目） | 作答 + 引用 |
| 项目资料-无权 | 滨江控规某指标 | 陈珊（无该项目权限） | 权限提示，**不泄露原文/引用** |
| 企业制度 | 报销/IT 流程 | 员工 | 作答 + 引用 |
| 超范围 | 投资测算 / CAD 解析 / 审批结论 | 任意 | 拒答（范围闸门） |
| 无依据 | 知识库未收录的问题 | 任意 | 拒答（依据闸门） |
| 角色越权 | 高于自身权限级的资料 | 员工问 L3 | 按 ACL 过滤，不可见 |

> 评测运行时按题目绑定的模拟账号解析身份（`resolveEvaluationUserId`，见对应单测），
> 保证「同一问题不同角色应有不同结果」可被验证。

## 4. 通过口径（Pass Criteria）

每道题的「期望」是一个**行为标签**，而非逐字匹配答案：

| 期望标签 | 判定通过的条件 |
| --- | --- |
| `作答` | `foundEvidence=true` 且引用非空（可附加：引用命中预期文件） |
| `拒答` | `foundEvidence=false` 且 refusalReason 属于范围/依据类 |
| `权限提示` | 返回无权文案，`citations` 为空，回答不含无权原文 |

**反例守恒**：「权限提示」必须与「无依据拒答」区分——前者是「有但你无权」，后者是「确实没有」。
混淆会让用户误判知识库完整性，属于必须拦截的错误（见 `research.md` 洞察 1）。

---

## 5. 指标

| 指标 | 口径 | 目标 |
| --- | --- | --- |
| 总通过率 | 通过题数 / 总题数 | 持续上升，改动不回退 |
| 拒答精确率 | 应拒答且正确拒答 / 全部拒答输出 | 高（防误拒有效问题） |
| 拒答召回率 | 应拒答且正确拒答 / 全部应拒答 | 高（防漏拒编造） |
| 越权泄露数 | 无权原文出现在回答/引用的次数 | **必须为 0** |
| 引用可溯源率 | 作答题中引用含文件/章节/页码的比例 | 100% |

> 越权泄露是「一票否决」指标：哪怕通过率很高，只要泄露 >0，本次改动不可发布。

## 6. 题库与导入导出

### 6.1 金标题库
内置金标题库见 `lib/rag/eval/goldenQuestions.ts`，运行器 `runEval.ts`；
真实 PDF 样本评测：`npm run eval:real-pdfs`。

### 6.2 导入（`lib/evaluation/importParser.ts`）
支持多格式，自动识别列（中文表头按特异性映射，`classifyHeader`）：
- CSV / TSV（含引号转义、列顺序无关、可无表头按位置解析）
- Markdown 表格（跳过分隔行）
- XLSX（2D 单元格，非字符串强制转换、丢弃空行）

### 6.3 导出（`lib/evaluation/exportResults.ts`）
导出 XLSX：表头与列定义一致，布尔/分数/指标格式化，默认文件名以 `.xlsx` 结尾
（`defaultEvaluationExportFilename`）。

## 7. 运行流程

```
导入题库 (/evaluation 上传 或 内置金标题库)
   → 逐题以绑定账号调用问答链路 (generateAnswer)
   → 对比 实际行为标签 vs 期望标签
   → 汇总通过率 + 分场景明细
   → 导出 XLSX 报告
   → 不通过项 → 定位 (检索/重排/拒答/权限) → 修复 → 重跑回归
```

---

## 8. 回归策略
1. **改动前先建基线**：任何检索/重排/拒答/权限改动前，先跑一次题库记录通过率与分场景明细。
2. **改动后对比**：通过率不得低于基线；越权泄露数必须保持 0。
3. **新 Bug 先补题**：发现的每个问答错误，先在题库/单测中加一条复现用例，再修——防止回归。
4. **单测同步**：涉及权限/解析/导入导出的行为变化，必须在 `tests/` 增补用例（当前 74 例）。

## 9. 与其它阶段的闭环
- 每条 [PRD](./PRD.md) 验收标准（AC-1~AC-6）→ 在本评测的场景矩阵中都有对应行。
- [需求调研](./research.md) 的关键洞察（无权≠无答案、表格即真相、信任来自可核对）→ 直接映射为通过口径与一票否决指标。
- 这使「写下的需求」与「验证的行为」严格对齐，避免文档与实现脱节。
