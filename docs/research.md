## 1. 用户画像（Persona）

角色与权限定义对齐 `lib/knowledge/permissions.ts`，内置 6 个模拟账号。

### P1 普通员工（`employee`，L1）
- **代表账号**：张明（规划一所，授权项目 `project-riverfront`）、陈珊（城市设计中心，授权 `project-industrial`）。
- **目标**：快速查到公开规范条文、指标；查看自己被授权项目的资料。
- **痛点**：不知道指标藏在哪份文件第几页；越权资料不该看到也不想误触。
- **高频问题**：「某用地容积率上限是多少」「报审需要哪些成果」。

### P2 项目负责人（`project_manager`，L2）

- **目标**：查 L2 非项目资料 + 自己负责项目的全部资料；管理本项目文档。
- **痛点**：跨项目资料不该串；需确认引用来自正确项目版本。

### P3 管理员（`admin`，L3）
- **代表账号**：王磊（知识管理部，**默认账号**）。
- **目标**：全量可见；上传/分类/配权限/上下架文档；跑评测看质量。
- **痛点**：要保证权限配置不出错、知识库质量可量化。

### P4 开发人员（`developer`，L3，当前对外隐藏）
- **代表账号**：赵工（数字化研发组）。
- **目标**：用 `/debug` 看三路检索得分与重排过程，调优检索。
- **现状**：开发工具与该账号在选择器中暂时隐藏（见单测 `开发工具入口暂时对所有角色隐藏` / `人员切换列表暂时隐藏开发人员账号`）。

## 2. 用户旅程（Journey）

### 旅程 A：员工查规范（成功路径）
1. 选择身份（默认管理员，可切员工） → 2. 输入问题 → 3. 系统按角色 ACL 过滤可见资料 →
4. 三路检索 + 重排 → 5. 返回【结论/依据/注意】+ 引用卡片（含表格切片）→ 6. 点引用核对原文。

**情绪曲线**：输入时存疑（怕编造）→ 看到可溯源引用后信任上升。

### 旅程 B：命中无权资料（权限拒答）
员工问到他人项目资料 → 检索命中但 ACL 不通过 → **只返回权限提示，不展示任何原文/引用**。
**关键体验**：不泄露 + 不让用户误以为系统没数据（区分「无权」与「无依据」）。

### 旅程 C：依据不足（诚实拒答）
问题超范围或命中片段不足以支撑 → 返回【无法确定/原因/建议】三段式，引导补充资料。
**关键体验**：宁可不答，不可编造。

### 旅程 D：管理员维护知识库
上传文档 → 配置分类/权限/项目归属 → 触发处理流水线 → 入库 → 跑 `/evaluation` 看通过率 → 按反馈迭代。

## 3. 竞品 / 方案对比

| 方案 | 可溯源 | 表格处理 | 检索时权限隔离 | 拒答机制 | 私有部署 |
| --- | --- | --- | --- | --- | --- |
| 通用大模型直接问答 | ✗ 易编造 | 弱 | ✗ | 弱 | 取决于服务 |
| 通用企业网盘 + 全文检索 | 部分 | 无结构 | 仅「能否打开」非「检索时」 | 无 | ✓ |
| 通用 RAG 框架（开箱） | ✓ | 一般（表格按文本切） | 需自建 | 需自建 | ✓ |
| **本产品** | ✓ 文件/章节/页码 | **表格一级对象+精确渲染** | **检索前过滤** | **范围+依据双闸门** | ✓（OpenAI 兼容/可 mock） |

**结论**：差异化集中在三点——可信抽取式回答、表格 RAG、检索时权限隔离。这三点正是设计院场景最在意、通用方案最薄弱的地方。

## 4. 需求优先级（MoSCoW）

| 级别 | 需求 |
| --- | --- |
| Must | 抽取式问答+引用、权限前置过滤、依据/范围拒答、文档处理流水线、评测闭环 |
| Should | 表格切片渲染、切片查看器、反馈记录 |
| Could | 检索调试可视化、扫描件 OCR/视觉识别（默认关） |
| Won't（本期） | 真实账号体系、持久化存储、CI 门禁 |

## 5. 关键洞察（指导设计）
1. **「无权」≠「无答案」**：必须在交互上区分两者，否则用户会误判知识库不全 → 体现在旅程 B/C 的不同文案。
2. **表格是规范的核心载体**：指标/退线/配建几乎都在表格 → 表格必须结构化，不能按纯文本切。
3. **信任来自可核对**：每条结论都要能点回原文页码 → 引用卡片是信任的承载体。
